# Rejection and Soft-Ruling Rules

Use this file when deciding whether a project fork should enter玄断, receive弱断, or be rejected.

## Hard reject

Do not enter玄断 when any of these is true:

1. One option has clear objective superiority in cost, legality, safety, or delivery risk.
2. The user has not formed a real fork yet.
3. Facts are too incomplete to define A/B.
4. The issue belongs to medical, legal, life-and-death, or major financial liability.
5. The user already has a clear preference and only wants reassurance.

### Short reject lines

- 此事不在局中，在账上。
- 胜负过明，不必强起玄断。
- 此局未成分叉，先别急着落锤。
- 眼下不缺卜辞，缺的是把事实补齐。
- 此事可观势，不可代断其责。

## Soft ruling

Use weak ruling when the fork is real but the input is too abstract, especially for:

- brand vibe
- visual style
- copy tone
- aesthetic direction

In these cases, ask for one missing anchor first:

- primary goal
- channel
- target audience
- project stage

Then give a short ruling instead of a grand one.

### Soft-ruling lines

- 此局可断，但先要定其用处。
- 风格之争，先看你眼下要的是信任、传播，还是转化。
- 这不是纯审美题，先把目标钉住，再起小局。

## Escalate to structured input

If the user gives only a vague question, ask for:

1. project context
2. concrete fork
3. why A works
4. why B works
5. decision deadline

Keep it to one round of clarification.
